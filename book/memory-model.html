Memory Model

Shared global memory

FuelVM has a new memory architecture where making calls between contracts is as easy as passing pointers to data, because call frames can read anything from a shared memory block. This is in contrast to the EVM where every contract call ahs to serialise data that is forwarded or returned - this is wasted processing which limits cross contract communication.

Everytime, including the first time, it creates a new instance of the EVM, and each instance of the EVM has access to its own separate memory, similar to RAM. This memory is distinct from any other instance and it has 2 other places for memory, calldata and return data. When you want to do a cross contract call or the first contract call, these instances dont have access to any other instance including the calldata or return. If you want to pass data from one contract to another, or from a transaction to the first contract call, you have to pass it in as that call data. And when you want to return data, you have to put it inside the return data. You cant use the memory.

The problem with this is that if you want to share data, you have to copy that data to calldata. Notably, calldata is a single, linear memory chunk. This means that if you have some sort of structure that is in 2 places of memory, you first havt to serialize it so its all in one place, and then copy it. First you must serialize everytime you want to move data from one EVM instance to another, and then you have to copy into calldata.

SImilarly, to access return data you have to copy from memory from return instance, and then put it in the return data. Return data is also linear memory which means you also have to serialize the data. These are wasteful operations.

In the FuelVM, instead of each instance of a contact call having its ownd educated memory, we have single shared block of memory that is shared across all the callframes. They can read from anywhere int his big ass chunk of memory. This means if you want to pass data from one callf rame to another, you dfont have to copy it, you just have ti pass a pointer to the data. Theres global readability, any callframe can read anything, and this includes the serialized transaction and contract bytecode, and all the data all live in this shared chunk of data. You don’t need special instructions like in ethereum. In EVM, you cant do a lot of manupiation od ata in call data so if you want to manipulate, you have to copy formmemory to call data and then to use it you have to copy form calldata to memory again. I.e. if you want to do hashing, they dont operate on calldata or return data, it does it o nmemory. So once to the buffer, then back to memory. We have script transactions that have script data which are equivalent to transaction data in EVM The nuance is that its not glovally writable, because you dont want a contract that you call to manipulate the data of the caller or something. You can only write to your own subset of the shared memory.
