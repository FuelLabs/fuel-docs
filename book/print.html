Technology

Parallel Transaction Execution

Fuel is designed as a UTXO-based design optimized for maximum scalability, allowing for parallelized transaction execution on consumer grade hardware.

One of the key features of a blockchain is that it is a state machine- one of its core functions is to keep track of current and past states, and the transitions between those states. Changes in users' balances, updates to the state of a variable in a smart contract, and the deployment of a new smart contract are all examples of changes of state.

In the UTXO model, each unspent transaction output is stored individually, as opposed to a single, final account balance for a given wallet. The movement of these UTXOs are tracked as a directed graph, and the entire graph is the state, as opposed to the account model bookkeeping mechanism where the state are the final balances of wallets.

The UTXO model allows for parallel transaction execution because you can process multiple UTXOs at the same time, because there are no dependencies on other UTXOs to create a global state. Read more on UTXOs here.

State Access Lists

UTXOâ€™s

There are two existing mechanisms for bookkeeping and managing state transitions in a blockchain, the account model and the UTXO model.

In the account model, the chain maintains and updates a database of the balances of every wallet as a global state. When a transaction is executed and changes the state, the account model updates this list of every account, or wallet, and makes the appropriate updates to reflect the transaction. If you imagine this as a simple list keeping track of balances by wallet, the final line would represent the global state.

[insert graphic here]

UTXO, an acronym for unspent transaction, is a model where the movemment of assets are recorded as a directed graph where the entire graph of transactions is the global state, as opposed to the account model where only the most recent update to the database is the global state. The UTXO model does not have this concept of wallets or accounts, only individual transactions called UTXOs.

Transaction outputs must always be spent as a whole, regardless of what fraction of the whole the payee wants to send in a transaction. This is where the term "unspent" comes from. Of the whole UTXO, there can be at most two transaction outputs. One containing the amount sent to the payee, and one containing the unspent amount which is sent to the original owner.

[Insert graphic here]

Example

  • User 0x99395039408603 has a balance of 100 BTC, which could be a single UTXO worth 100 $BTC, or a combination of UTXOs that add up to 100 $BTC.
  • The user sends 12 $BTC to their friend, 0x883952938231. This creates two outputs:
    • One output of 12 $BTC to the payee, 0x883952938231
    • One output of the unspent 88 $BTC to the payer.

Because there is no concept of wallets in the UTXO model, the balances a user sees in their wallet are calculated on the clientside by summing all of the UTXOs that user is in control of.

Memory Model

Shared global memory

FuelVM has a new memory architecture where making calls between contracts is as easy as passing pointers to data, because call frames can read anything from a shared memory block. This is in contrast to the EVM where every contract call ahs to serialise data that is forwarded or returned - this is wasted processing which limits cross contract communication.

Everytime, including the first time, it creates a new instance of the EVM, and each instance of the EVM has access to its own separate memory, similar to RAM. This memory is distinct from any other instance and it has 2 other places for memory, calldata and return data. When you want to do a cross contract call or the first contract call, these instances dont have access to any other instance including the calldata or return. If you want to pass data from one contract to another, or from a transaction to the first contract call, you have to pass it in as that call data. And when you want to return data, you have to put it inside the return data. You cant use the memory.

The problem with this is that if you want to share data, you have to copy that data to calldata. Notably, calldata is a single, linear memory chunk. This means that if you have some sort of structure that is in 2 places of memory, you first havt to serialize it so its all in one place, and then copy it. First you must serialize everytime you want to move data from one EVM instance to another, and then you have to copy into calldata.

SImilarly, to access return data you have to copy from memory from return instance, and then put it in the return data. Return data is also linear memory which means you also have to serialize the data. These are wasteful operations.

In the FuelVM, instead of each instance of a contact call having its ownd educated memory, we have single shared block of memory that is shared across all the callframes. They can read from anywhere int his big ass chunk of memory. This means if you want to pass data from one callf rame to another, you dfont have to copy it, you just have ti pass a pointer to the data. Theres global readability, any callframe can read anything, and this includes the serialized transaction and contract bytecode, and all the data all live in this shared chunk of data. In EVM, you cant do a lot of manupiation od ata in call data so if you want to manipulate, you have to copy formmemory to call data and then to use it you have to copy form calldata to memory again. I.e. if you want to do hashing, they dont operate on calldata or return data, it does it o nmemory. So once to the buffer, then back to memory. We have script transactions that have script data which are equivalent to transaction data in EVM The nuance is that its not glovally writable, because you dont want a contract that you call to manipulate the data of the caller or something. You can only write to your own subset of the shared memory.
